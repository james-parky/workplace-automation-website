Resource Allocation and Skill/Requirement Based Matching

Get Started

Background

UCL’s Computer Science department has recently been transitioning a number of day to day activities in the form of business transformation as workflows. In partnership with Microsoft, a number of tools and systems have been requested across the teaching and learning domains.

The two highest burdens faced by academics and research students is the areas of final year projects matching (160 individual students to be matched to projects) and in TA allocation of classes (over 100 classes, in category types such as labs, tutorials, Q&A sessions, project supervision).

Project

  • Projects Allocation of Academic Staff for Final Year Students and advertising research areas/projects
  • TA Allocation of PhD Students for classes and advertising classes

Technologies

Our project is powered by the follwing Microsoft technologies, allowing us to create a tailored solution to the problem we've been presented while incorporating smooth integration with exisiting UCL infrastructure.

Dataverse

Dataverse lets you securely store and manage data that's used by business applications. Standard and custom tables within Dataverse provide a secure and cloud-based storage option for your data.

Project Operations

Project Operations connects sales, resourcing, project management, and finance teams in a single application to win more deals, accelerate project delivery, and maximize profitability.

Power Automate

Power Automate is a service that helps you create automated workflows between your favorite apps and services to synchronize files, get notifications, collect data, and more.

Power Apps

Power Apps is a suite of apps, services, and connectors, as well as a data platform, that provides a rapid development environment to build custom apps for your business needs.

Data Privacy

When developing a website, it is very important that you do not accidentally display sensitive data that would violate GDPR regulations.

As a failsafe, it's a great idea to use exclusively fake data right up until your project is ready for production.

Often, you will need different data to be shown to different people. For example, in our project our two user types (students and professors) should be given access to very different sets of data. To ensure this, we used web roles and table permissions to hide or show information based on user type.

Liquid

Liquid is a template language, written amongst HTML that uses a combination of objects, tags, and filters inside template files to display dynamic content. Liquid, like any template language, creates a bridge between an HTML file and a data store — in our context, the data we use for matching.

Liquid can be learnt fairly quickly through the use of the official documentation. Here are a few examples of how we used Liquid in our project:
    Hiding Sensitive Data During Development
    • At times during our development, we needed to display data that should be available to authenticated users only. To do this, we used Liquid control flow tags to show or hide data based on this condition:
    •    {% if user %}
    •      {% include 'my_page' %}
    •    {% endif %}

    • If the user is not signed in, 'user' will not exist, meaning the variable holds a value of 'Nil'. This means the 'user' variable is 'falsy', causing the condition to be evaluated as false and therefore not display my_page.

    • It should be noted that this is not the most sound and elegant solution to hide content from the unauthorised. PowerApps offers mechanisms to acheive this through Web Roles.

    Displaying User-Specific Information
    • It's often useful to have a webpage display different content depending on who the user is, and what information we have about them. Objects may have certain attributes which can be used to acheive this. As a simple example, we used:
    •   {{ user.fullname }}
    • to display a header which display the correct name for the user that's currently logged in.

    Iteration to Display Elements in an Array
    • Under different conditions, different content may need to be included in the header. We used Liquid to display all headings which should be displayed within a given Web Link Set. Here is a heavily simplified method with which we did this:
    •   {% for link in primary_nav.sublinks %}
    •     {% unless forloop.first %}
    •       <div style="vertical-divider"></div>
    •     {% endunless %}
    •     {{ link.name }}
    •   {% endfor %}

Portal Structure

You could describe a portal as a conglomeration of web pages, data structures, and automated flows such that users may easily interact with data relevant to themselves. You could describe the top-down structure of the portal as follows:

Portal → Web Pages → Web Templates → Page Templates → HTML, Liquid, Content Snippets

To elaborate; a Portal may be thought to be a directed graph, where each node is a webpage and edges are a redirection, such as clicking a button to reach a different page.

To display content on a webpage, you must associate the page with a particular page template. Where a web page is essentially an exact URL, a page template may be described as a blueprint for how the content of the page will be displayed. Within this page template, you may write the actualy content of the page, through some combination of HTML, Liquid, and JavaScript.
